## 1. Продукт и цель

**PM Lite** — онлайн-сервис управления проектами для SMB-организаций проектного типа.

**Фокус:** прозрачность статусов проектов, контроль загрузки команды, простой контроль сроков.

**Принцип MVP:** не прототип — реально рабочий инструмент, на котором уже можно вести проекты и параллельно проверять гипотезы ценности. Любое решение оценивается через: «Можно ли на этом уже вести реальный проект?»

**Особенности реализации:**
- JWT payload: `{ sub (userId), orgId, isAdmin, roleId }` — хранится в localStorage
- axios-interceptor автоматически добавляет JWT в заголовки
- `qc.clear()` вызывается при логине И при логауте (сброс кэша между сессиями)
- React Query staleTime: 30 000 ms
- Демо-база использует фиксированные UUID для org и admin, чтобы JWT не инвалидировались после ресида

## 3. Регистрация, пользователи и доступ

### 3.1 Регистрация и онбординг (Эпик A1) — ✅ реализовано

#### Уточняющие вопросы и решения

| Вопрос | Решение |
|--------|---------|
| Как регистрируется организация? | Форма: название орг, e-mail, пароль, имя. Создаётся орг + admin-пользователь + 5 системных ролей. |
| Подтверждение e-mail? | Да — HMAC-подписанная ссылка в URL, без отдельной таблицы. Срок: 24 ч. |
| Как пригласить пользователя? | Admin вводит e-mail + выбирает роль из org_roles → создаётся запись `invitations` → ссылка на почту. |
| Самостоятельная регистрация? | Да — пользователь вводит slug организации + свои данные. Попадает в pending, admin одобряет. По умолчанию — роль «Исполнитель». |
| Мультиорг? | 1 e-mail = 1 организация; переключение контекстов не нужно. |

#### Реализованные user stories

| US | Описание | Статус |
|----|---------|--------|
| US-A1.1 | Регистрация организации + email verification | ✅ |
| US-A1.2 | Admin приглашает пользователя по e-mail с выбором роли | ✅ |
| US-A1.3 | Самостоятельная регистрация по коду (slug) орг, статус pending, одобрение admin | ✅ |

#### Технические детали A1
- Верификационный токен: `base64url(JSON{sub,exp}).HMAC-SHA256` — без отдельной таблицы
- JWT payload: `{ sub, orgId, isAdmin: boolean, roleId: string|null }` — localStorage
- Slug: русские символы транслитерируются в латиницу перед нормализацией
- Email (dev): SMTP localhost:1025 (MailHog); ошибки — логируются, приложение не падает

### 3.2 Роли

| Роль | Описание |
|------|---------|
| **Администратор** (isAdmin=true) | Полный доступ ко всему. Хранится флагом `is_admin`, не через org_roles. |
| **Руководитель проекта (РП)** | Создаёт проекты, управляет задачами и исполнителями, видит загрузку, использует AI |
| **Руководитель подразделения** | Видит все задачи подчинённых (матричная оргструктура) |
| **Администратор проекта** | Управляет конкретным проектом, но не создаёт новые |
| **Исполнитель** | Видит только свои задачи, может менять статус только своих |
| **Наблюдатель / Стейкхолдер** | Только просмотр |

#### Уточняющие вопросы о модели ролей (сессия 2026-06-23)

| Вопрос | Ответ |
|--------|-------|
| Фиксированные роли (в коде) или кастомные (через UI)? | **Кастомные** — Admin настраивает через страницу «Роли и права» |
| Какие роли нужны из коробки? | 5 системных (isSystem=true): РП, Рук. подразделения, Адм. проекта, Исполнитель, Наблюдатель |
| Можно ли удалить системную роль? | Нет. Удаляются только кастомные без назначенных пользователей |
| Оргструктура матричная? | Да — РП видит всех исполнителей проекта; Рук. подразделения видит задачи своих подчинённых |
| Делегирование? | РП делегирует проект/часть плана другому; делегированный РП видит только свою часть |

#### Техническая реализация

- Таблицы: `org_roles` (id, org_id, name, description, is_system) + `role_permissions` (role_id PK, permission PK)
- `User.isAdmin` (bool) — admin получает все права автоматически, без org_role
- `User.orgRoleId` (UUID nullable) — ссылка на org_roles
- При регистрации орг: автоматически создаются 5 системных ролей (`seedDefaultRoles`)
- Middleware `authenticate.ts`: загружает `Set<PermissionKey>` из БД (только для не-admin)
- `requirePermission(key)` — фабрика preHandler; `requireAdmin` — проверяет `isAdmin`  
- API: `GET /api/roles/meta`, `GET/POST/PATCH/DELETE /api/roles`, `PATCH /api/users/:id/role`  
- Frontend: страница `/dashboard/roles` — список ролей + форма с чекбоксами по группам

### 3.4 Управление пользователями (Эпик A2) — ✅ реализовано · 🤖 автотест: компиляция OK

#### US-A2.1 — Список сотрудников

**Требования:**
- T1: Таблица пользователей: имя, e-mail, роль, статус (активен / ожидает одобрения)
- T2: Фильтр по роли и статусу

**Решения:**
- Список: `GET /api/users` — включает `dailyCapacityHours`, `orgRoleName`
- Сортировка: активные перед неактивными, затем по дате создания
- Фильтрация клиентская (команда обычно < 100 чел.; лишний round-trip не нужен)
- Ожидающие одобрения — отдельный блок с жёлтой меткой «Ожидает»
- Счётчик «3 из 7» рядом с заголовком, когда фильтр сужает выборку
- Кнопка «Сбросить» появляется при любом активном фильтре

#### US-A2.2 — Смена роли и деактивация

**Требования:**
- T1: Смена роли
- T2: Деактивация — пользователь теряет доступ, данные остаются в БД

**Решения:**
- `PATCH /api/users/:id/role` — body `{ orgRoleId: uuid | null }`
- `DELETE /api/users/:id` — только `isActive = false`; физического удаления нет
- Смена роли: `<select>` с автосохранением на `onChange` (без кнопки «Сохранить»)
- Защита: нельзя изменить или деактивировать самого себя (400)
- Нельзя назначить роль из чужой организации (400)
- Для `isAdmin`-пользователей и ожидающих одобрения: роль отображается текстом (без select)

#### US-A2.3 — Рабочая доступность

**Требования:**
- T1: Настройка рабочих часов в день на каждого сотрудника
- T2: Добавление периодов отсутствия (отпуск, больничный, другое)
- T3: Данные отсутствий доступны для timeline загрузки (Эпик PM3)
- **AC:** При добавлении отсутствия — подсветить незавершённые задачи сотрудника, дедлайн которых попадает в период

**Решения по T1 (часы/день):**
- `PATCH /api/users/:id/capacity` — body `{ dailyCapacityHours: 0.5..24 }`
- Поле `daily_capacity_hours` в таблице `users` (Decimal 4,2; дефолт 8)
- Сохранение по кнопке «Сохранить» (не автосохранение — изменение влияет на расчёты загрузки)

**Решения по T2 (отсутствия):**
- `GET /api/users/:id/absences` — список, сортировка по `startDate`
- `POST /api/users/:id/absences` — body `{ type, startDate, endDate, notes? }`; возвращает `{ absence, affectedTasks }`
- `DELETE /api/users/:id/absences/:absenceId`
- Типы: `vacation` (Отпуск), `sick` (Больничный), `other` (Другое)
- Валидация: `endDate >= startDate` (400 иначе)
- Все эндпоинты: `requireAdmin` (только Администратор)

**Решения по AC (affectedTasks):**
- Критерий: `assigneeId = userId`, `dueDate BETWEEN startDate AND endDate`, `status NOT IN ('done','cancelled')`
- Возвращается при POST вместе с созданным отсутствием
- UI: жёлтый блок-предупреждение со списком задач + проект + дедлайн; закрывается крестиком

**UI — `UserAvailabilityPanel`:**
- Открывается кнопкой-иконкой 📅 в строке сотрудника (только admin, только активные)
- Accordion — инлайн под строкой; одновременно открыта одна панель на всю страницу
- Форма добавления отсутствия — inline внутри панели, без модалки
- Цвета бейджей: синий (отпуск), красный (больничный), серый (другое)

## 3b. Настройки организации (A3) — ✅ реализовано · 🤖 автотест: API OK

### US-A3.1 Логотип организации

**Требование:** Администратор загружает логотип организации; он отображается в интерфейсе и в отчётах.

**Решения:**
- Формат передачи: base64 dataURL в JSON-теле — без `@fastify/multipart`, достаточно для MVP
- Максимальный размер: 2 МБ (проверка на фронтенде до FileReader)
- Хранение: поле `logo_url TEXT` в таблице `organizations` (уже было в схеме)
- Допустимые типы: любой `image/*` (проверяется через `file.type`)
- Предпросмотр до сохранения: FileReader → state `preview`; кнопка «Сохранить» активна только при preview
- Кнопка «Удалить логотип» обнуляет `logo_url` через `DELETE /api/org/logo`

**API:**
```
GET    /api/org           — инфо об организации (name, slug, logoUrl, planType)
PUT    /api/org/logo      — { logoDataUrl: string } → обновить (admin)
DELETE /api/org/logo      — удалить (admin)
```

### US-A3.2 Кастомные статусы проектов

**Требование:** Администратор создаёт, редактирует, удаляет кастомные статусы проектов (название, цвет); меняет порядок drag-and-drop.

**Решения:**
- T1 CRUD: название (строка ≤ 100), цвет (hex `#RRGGBB`, color-picker)
- T2 Порядок: HTML5 drag-and-drop без сторонних библиотек; при drop → `PUT /api/org/project-statuses/reorder`
- Уникальность: `UNIQUE (organization_id, name)` → 409 при дублировании имени
- Удаление блокируется если статус используется в проектах → ошибка: «Статус используется в N проект(ах). Сначала смените статус проектов.»
- `sort_order` при создании = `max(sort_order) + 1`; reorder — транзакция по индексам
- Маршрут `PUT /reorder` зарегистрирован **до** `PATCH /:id` во избежание коллизий

**API:**
```
GET    /api/org/project-statuses          — список по sort_order
POST   /api/org/project-statuses          — создать { name, color } (admin)
PATCH  /api/org/project-statuses/:id      — изменить { name?, color? } (admin)
DELETE /api/org/project-statuses/:id      — удалить; блокируется если используется (admin)
PUT    /api/org/project-statuses/reorder  — { ids: string[] } в новом порядке (admin)
```

### US-A3.3 Справочник клиентов

**Требование:** Администратор ведёт справочник клиентов; проекты группируются и фильтруются по клиенту.

**Решения:**
- T1 CRUD: поля — название* (обязательное ≤ 200), контактные данные (≤ 500), комментарий (textarea ≤ 2000)
- Удаление клиента **не** удаляет проекты: `client_id FK ON DELETE SET NULL`
- Сортировка: `createdAt ASC`
- UI: раскрываемые строки — контакт и заметки скрыты под chevron; одновременно раскрыта одна строка

**API:**
```
GET    /api/org/clients     — список клиентов организации
POST   /api/org/clients     — создать { name, contactInfo?, notes? } (admin)
PATCH  /api/org/clients/:id — изменить { name?, contactInfo?, notes? } (admin)
DELETE /api/org/clients/:id — удалить; проекты не удаляются, client_id → NULL (admin)
```

| Что происходит после успешного сброса пароля? | Автоматический вход — сервер возвращает JWT, пользователь сразу попадает в приложение |
| UX кнопки выхода: немедленный logout или dropdown? | Dropdown-меню при клике на аватар (пункт «Выйти»; в будущем — «Профиль») |

### Детали реализации

**US-S1.1 — Вход:**
- `POST /api/auth/login` — проверка email + пароль, возврат JWT (7 дней)
- Ошибка 401 при неверных данных; 403 если аккаунт не активирован
- Фронтенд: `LoginPage.tsx` — форма, Zod-валидация, JWT → localStorage через Zustand
- Ссылка «Забыли пароль?» рядом с лейблом поля пароля (prop `labelRight` добавлен в `FormField`)

**US-S1.2 — Восстановление пароля:**
- `POST /api/auth/forgot-password`:
  - Удаляет старые токены пользователя (только один активный одновременно)
  - Создаёт `PasswordResetToken` (token генерируется в БД: `encode(gen_random_bytes(32), 'hex')`)
  - Отправляет email со ссылкой `APP_URL/reset-password?token=...`
  - Всегда возвращает 200 (защита от перебора пользователей)
- `POST /api/auth/reset-password`:
  - Проверяет токен: существует + `usedAt IS NULL` + `expiresAt > now()`
  - Обновляет `passwordHash` и помечает токен `usedAt = now()` в одной транзакции
  - Возвращает JWT для автологина
- Фронтенд:
  - `ForgotPasswordPage.tsx` — форма ввода email + экран «письмо отправлено»; маршрут `/forgot-password`
  - `ResetPasswordPage.tsx` — поля «новый пароль» + «повторите» (min 8 символов, Zod refine) → `setAuth()` + редирект на `/projects`; маршрут `/reset-password`

**US-S1.3 — Выход:**
- `clearAuth()` — удаляет JWT из localStorage, обнуляет Zustand-стор
- Фронтенд: `UserMenu` в `AppLayout.tsx` — dropdown при клике на аватар-кружок; закрывается кликом вне меню (`mousedown` на `document`)
- После выхода — редирект на `/login`

### Схема БД

```
password_reset_tokens — (id UUID, user_id UUID FK→users, token TEXT UNIQUE,
                          expires_at TIMESTAMPTZ, used_at TIMESTAMPTZ?, created_at TIMESTAMPTZ)
  token: генерируется в БД encode(gen_random_bytes(32), 'hex')
  TTL: 1 час, устанавливается явно: Date.now() + 3_600_000
```

---

### 8.1 Уточняющие вопросы и согласованные ответы

#### Загрузка и права

**Q: Кто может загружать документы к проекту?**
A: Все роли. Исполнитель — только к своим задачам (где он назначен).

**Q: Кто может удалять документы?**
A: Автор + Admin. Soft delete (deletedAt).

**Q: Нужны ли статусы документов (черновик / согласован / утверждён)?**
A: В MVP нет. Только загрузка и хранение.

**Q: Нужны ли уведомления при загрузке/обновлении документа?**
A: В MVP нет.

#### Версионирование

**Q: Как реализовать версионирование?**
A: Вручную — кнопка «Загрузить новую версию». Старые версии сохраняются. Текущая версия — последняя по versionNumber DESC.

**Q: Нужен ли currentVersionId FK на документе?**
A: Нет — создаёт циклическую зависимость FK. Текущая версия вычисляется ORDER BY versionNumber DESC, LIMIT 1.

#### Матрица доступа

**Q: Как настраивается матрица доступа роль x тип документа?**
A: Admin задаёт org-умолчания. PM переопределяет на уровне своего проекта.

**Q: Приоритет правил при конфликте?**
A: Специфичность: (org+project+type) → (org+project+NULL) → (org+type) → (org+NULL) → default: canView=true, canUpload=false.

**Q: Что видит Исполнитель по умолчанию?**
A: Видит все типы, кроме финансовых (категория 'financial'). Загружает только к своим задачам.

**Q: NULL в уникальном индексе матрицы — как избежать коллизий?**
A: PostgreSQL рассматривает каждый NULL как уникальный. Управление через replace-all: удалить все правила для scope → создать новые одним createMany.

#### Хранилище

**Q: Два режима хранения — разные инстансы или одно приложение?**
A: Одно приложение PM Lite, режим выбирается per-organization через StorageProviderSetting.

**Q: Как работает внешнее хранилище (external)?**
A: Клиент устанавливает плагин на своей инфраструктуре. PM Lite вызывает REST API плагина для upload/download/delete. Файлы физически на стороне клиента. PM Lite хранит только метаданные + external_id.

**Q: Кто пишет плагин?**
A: Мы. Поставляем и спецификацию, и референсную реализацию (plugin/).

**Q: Как хранить API-ключ плагина?**
A: SHA-256 хэш в pluginApiKeyHash. В MVP хэш используется как Bearer-токен при вызовах плагина.

**Q: Как проверить доступность плагина?**
A: Кнопка «Проверить подключение» в UI → POST /api/storage-settings/verify → GET /api/v1/health на плагине → запись pluginVerifiedAt + pluginVersion.

**Q: Cloud хранилище — S3 или файловая система?**
A: В MVP — файловая система. Путь: UPLOADS_DIR/<orgId>/<docId>/<versionId>_<fileName>. Checksum: SHA-256.

#### Полиморфный parent

**Q: Документ может быть привязан к проекту, задаче или клиенту — как реализовать FK?**
A: Полиморфный паттерн: parent_type: 'project'|'task'|'client' + parent_id: UUID (без FK в БД). Плюс денормализованный project_id для быстрой фильтрации по проекту.

#### Прочие

**Q: Нужен ли workflow согласования документов?**
A: В MVP нет.

**Q: Что происходит с документами при архивировании проекта?**
A: Остаются. Доступны через общую библиотеку.

**Q: Как скачивать файлы, если endpoint защищён JWT (браузерский тег a href не шлёт заголовок)?**
A: Программный fetch с Authorization header → Blob URL → click на временный anchor (utils/downloadFile.ts).

  - Захват правого края → изменение длительности (mode: 'resize') → обновление dueDate
  - Обновление оптимистичное (немедленно на UI), затем PATCH /api/tasks/:id

**Зависимости между задачами:**
- Типы: FS (Finish-Start), SS (Start-Start), FF (Finish-Finish), SF (Start-Finish)
- Атрибут: lagDays (отставание в днях)
- Отображение: SVG-стрелки с ломаными линиями и наконечниками
- Цвет: серый (обычная) / красный (на критическом пути)

**Критический путь:**
- Вычисляется через полный CPM (Kahn's топологическая сортировка + прямой проход ES/EF + обратный LS/LF)
- Total Float = LS - ES; если ≤ 0 → задача критическая
- Красный цвет: бары критических задач и стрелки между ними
- Включается/выключается чекбоксом в тулбаре

**Базовый план:**
- Кнопка «Зафиксировать базовый план» → записывает startDate→baselineStart, dueDate→baselineFinish для всех задач
- Чекбокс «Показать базовый план» → отображает серую полоску под баром

### 9.3 Карточка задачи из Gantt
- **Одиночный клик** на строку/бар → выбор задачи (для хоткеев)
- **Двойной клик** на строку или бар → открывает TaskDrawer (правая панель)

users                  — пользователи (id, orgId, email, fullName, isAdmin, orgRoleId)
org_roles              — роли в организации (с набором прав PermissionKey)
project_role_members   — участники проекта с ролью
clients                — справочник клиентов
projects               — проекты (id, orgId, clientId, name, status, pmId, budget, isArchived)
tasks                  — задачи (id, projectId, parentId, title, status, priority, taskType,
                                  startDate, dueDate, percentComplete, effortHours,
                                  baselineStart, baselineFinish, sortOrder)
task_dependencies      — зависимости (taskId, predecessorId, type:FS/SS/FF/SF, lagDays)
board_columns          — колонки Канбана per-project
task_comments          — комментарии к задачам
task_role_effort       — план часов по роли (taskId, roleId, totalHours, distributionRule)
task_effort_periods    — разбивка по неделям (taskRoleEffortId, weekStart, plannedHours)
time_entries           — фактические часы (userId, taskId, date, hours)
password_reset_tokens  — токены сброса пароля (userId, token unique, expiresAt, usedAt)
project_templates      — шаблоны проектов
template_tasks         — задачи шаблона с formulaJson
template_dependencies  — зависимости в шаблоне
norm_params            — параметры нормирования
documents              — документы (parent_type/parent_id полиморф., project_id денорм.)
document_versions      — версии документов
document_types         — типы документов (системные + кастомные)
document_access_rules  — матрица доступ (org/project × role × type)
storage_provider_settings — настройки хранилища (cloud/external)
