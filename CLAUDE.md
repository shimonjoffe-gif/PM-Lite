# PM Lite — Правила работы на проекте

## Стек и окружение

- **Backend:** Node.js + TypeScript + Fastify 4.x + Prisma + Zod — порт 3000
- **Frontend:** React 18 + TypeScript + Vite + Tailwind + TanStack Query — порт 5173
- **БД:** PostgreSQL 16, пользователь `pm_lite`, БД `pm_lite`
- **Демо-логин:** `admin@pmgroup.ru` / `Demo1234!` (все 11 пользователей)
- **Документация:** `FTT.md` (требования), `BACKLOG.md` (баги и беклог), `QUICKSTART.md` (запуск)

---

## Правило 1 — Уточнение задачи перед реализацией

Перед началом любой реализации:

1. Задавай все уточняющие вопросы — **по одному**, с вариантами ответов (AskUserQuestion).
2. После получения всех ответов — проверь, не осталось ли неясностей, задай дополнительные вопросы если нужно.
3. Сформируй **резюме задачи**: что делаем, почему, какой результат ожидается.
4. Кратко опиши **план реализации**: какие файлы затронешь, в каком порядке.
5. Только после этого приступай к коду.

---

## Правило 2 — Завершение задачи

После завершения реализации или исправления бага **всегда без отдельной просьбы**:

1. Перезапусти бэкенд и фронтенд.
2. Убедись что оба порта (3000 и 5173) слушают.
3. Выведи:
   ```
   ✅ Готово: <краткое описание что сделано>

   Бэкенд:  http://localhost:3000
   Фронтенд: http://localhost:5173

   Логин:   admin@pmgroup.ru
   Пароль:  Demo1234!
   ```

**Перезапуск бэкенда:**
```powershell
# Остановить старый процесс
$p = Get-NetTCPConnection -LocalPort 3000 -ErrorAction SilentlyContinue | Select-Object -ExpandProperty OwningProcess
if ($p) { Stop-Process -Id $p -Force }
# Запустить новый
Start-Process powershell -ArgumentList "-NoExit", "-Command", "cd 'C:\YandexDisk\Обучение\Вайб_кодинг\PM Lite\backend'; npm run dev"
```

**Перезапуск фронтенда:**
```powershell
$p = Get-NetTCPConnection -LocalPort 5173 -ErrorAction SilentlyContinue | Select-Object -ExpandProperty OwningProcess
if ($p) { Stop-Process -Id $p -Force }
Start-Process powershell -ArgumentList "-NoExit", "-Command", "cd 'C:\YandexDisk\Обучение\Вайб_кодинг\PM Lite\frontend'; npm run dev"
```

> ⚠️ При перезапуске **никогда не выполнять** `db:seed-demo`, `db:migrate` или любые другие команды, меняющие данные в БД. Перезапуск — только процессы, данные в базе сохраняются.

## Правило 3 — Ручное тестирование

После перезапуска проводи тестирование **пошагово**:

1. Опиши шаг — одно конкретное действие.
2. Дождись подтверждения от пользователя («ок», «работает», скриншот).
3. Только после подтверждения переходи к следующему шагу.
4. В конце спроси: «Есть вопросы или что-то не работает?»

---

## Правило 4 — Фиксация багов

Если в ходе тестирования выявлен баг — **сразу** записывай в `BACKLOG.md`:

```markdown
| # | Описание | Шаги воспроизведения | Ожидалось | Фактически | Статус |
|---|----------|---------------------|-----------|------------|--------|
| N | ... | ... | ... | ... | 🔴 Открыт |
```

---

## Правило 5 — Дополнительные требования

Если в ходе работы появляется что-то, что не было в исходном запросе:

1. Спроси у пользователя: **«Это баг или новое требование?»**
2. **Баг** → записать в `BACKLOG.md`.
3. **Новое требование** → записать в `BACKLOG.md` (в раздел беклога) **и** в `FTT.md` (в соответствующий эпик).
4. Не реализовывать доп. требования молча — только после явного подтверждения.

---

## Правило 6 — Статусы задач в беклоге

Каждая задача (баг или новое требование) в `BACKLOG.md` имеет статус. Обновляй его при каждом переходе:

| Статус | Смысл |
|--------|-------|
| 🔵 Проработка | Задача зафиксирована, уточняются требования |
| 🟡 Реализация | Задача взята в работу, код пишется |
| 🟠 Тестирование | Реализация завершена, проверяется вручную |
| ✅ Закрыто | Протестировано и подтверждено пользователем |

**Правило обновления:**
- При старте уточнений → `🔵 Проработка`
- При начале кодирования → `🟡 Реализация`
- После перезапуска и начала тестирования → `🟠 Тестирование`
- После явного подтверждения пользователя → `✅ Закрыто`

---

## Соглашения по коду

- Комментарии в коде только если логика неочевидна.
- Не добавлять фичи сверх запрошенного.
- Обработка ошибок только на реальных границах (API, ввод пользователя).
- Структура модуля: `module.routes.ts` / `module.service.ts` / `module.schemas.ts`.
- После изменений в Prisma-схеме — всегда `npx prisma generate`.
